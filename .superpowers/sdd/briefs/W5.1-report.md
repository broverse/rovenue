### Fix 1 — Remove devDependencies from production Docker runtime image

**Problem:** The previous Dockerfile (`COPY --from=builder /app /app`) shipped ALL workspace node_modules to the runtime image, including tsx, vitest, testcontainers, and typescript — widening attack surface contrary to the F12 goal. The `CMD` correctly ran `node` (not tsx), but the packages remained on disk.

**Root cause of previous approach's failure:** `pnpm prune --prod` in a pnpm workspace removes root-level devDeps but keeps devDeps of all workspace packages (sdk-rn has vitest/tsx; api has tsx/vitest/testcontainers). The prune in the builder also lacked the pnpm store mount so it silently did nothing (0.4s). Neither approach worked reliably.

**Fix applied:** Restructured to use `pnpm deploy --prod`:
1. `deps` stage: `pnpm install --frozen-lockfile --ignore-scripts` (full workspace install, `--ignore-scripts` prevents fumadocs-mdx and other workspace postinstalls that need source files not copied in this stage)
2. `builder` stage: compile workspace packages (`@rovenue/shared`, `@rovenue/db`, `@rovenue/email-templates`) + API (`@rovenue/api`), then patch those workspace packages' `package.json` to redirect `main`/`exports` from `src/*.ts` → `dist/*.js` so that bare `node` (not `tsx`) can load them at runtime
3. `pnpm --filter @rovenue/api deploy --prod --ignore-scripts /app/prod`: creates self-contained directory with ONLY production node_modules (707 packages vs 2477 in full install)
4. Post-deploy: `rm -rf /app/prod/node_modules/vitest` — removes vitest which pnpm deploy included because better-auth declares it as an optional peer dep (it is NOT imported by the API at runtime)
5. `runtime` stage: `COPY --from=builder /app/prod /app` — only the lean prod output

Also upgraded base image `node:20-alpine` → `node:22-alpine` (satisfies `engines.node >=22.7` in package.json; the prior node:20 image caused `ERR_PNPM_UNSUPPORTED_ENGINE` failures). Pinned pnpm to 9.15.9 matching `packageManager` field.

## W5.2 Follow-up — tsup bundle fix (ESM ERR_MODULE_NOT_FOUND)

**Date:** 2026-06-21
**Triggering problem:** `tsc` with `moduleResolution: Bundler` emits extensionless relative
imports (`import { app } from "./app"`) that Node ESM cannot resolve at runtime, causing
`ERR_MODULE_NOT_FOUND` on `node dist/index.js` boot.

### tsup config (`apps/api/tsup.config.ts`)

```ts
import { defineConfig } from "tsup";

export default defineConfig({
  entry: ["src/index.ts"],
  format: ["esm"],
  platform: "node",
  target: "node22",          // matches Dockerfile FROM node:22-alpine
  bundle: true,
  noExternal: [/^@rovenue\//],  // inline workspace packages (@rovenue/db, @rovenue/shared, @rovenue/email-templates)
  external: [
    "pino", "pino-pretty", "pino-std-serializers",
    "pg", "bcryptjs",          // CJS packages with dynamic require() — Node 22 handles CJS↔ESM natively
  ],
  outDir: "dist",
  clean: true,
  sourcemap: true,
  esbuildOptions(options) {
    options.jsx = "automatic"; // for @rovenue/email-templates React components
  },
});
```

**Why `noExternal`:** `@rovenue/db`, `@rovenue/shared`, and `@rovenue/email-templates` have
`main: ./src/index.ts` (TypeScript source) and extensionless ESM dist output — neither is
loadable by Node at runtime. Inlining them into the bundle eliminates the problem.

**Why explicit `external` list:** When tsup inlines `@rovenue/shared` (which depends on pino),
it attempted to also bundle pino (a CJS package). tsup's `__commonJS` shim does not support
dynamic `require()` of Node built-ins (e.g., `require('os')` inside pino), throwing
`Error: Dynamic require of "os" is not supported`. Marking these CJS packages explicit-external
leaves them as `import` statements in the bundle; Node 22 handles CJS↔ESM interop natively
for installed npm packages. `pg` and `bcryptjs` have the same pattern (CJS with native addons
or dynamic requires).

### Build script change (`apps/api/package.json`)

```json
"build": "tsup && tsc --noEmit",
"typecheck": "tsc --noEmit",
```

`tsup` produces the runnable bundle. `tsc --noEmit` runs immediately after to preserve
type-checking in CI (same step, fails the build if there are type errors). A separate
`typecheck` script is also added for standalone invocation.

### Output path

`tsup` entry `src/index.ts` → `dist/index.js`. Matches Dockerfile `CMD ["node", "/app/dist/index.js"]`
and `"start": "node dist/index.js"` in package.json. No Dockerfile changes needed.

## W5.2 Final Fix — Ship email-templates locales into prod image

**Date:** 2026-06-21  
**Triggering problem:** After the tsup bundle fix, `node /app/dist/index.js` in the Docker image
crashed at startup with `ENOENT: no such file or directory, scandir '/app/locales'`. The
`@rovenue/email-templates` i18n module does `readdirSync(join(here, '..', 'locales'))` at
import time where `here = fileURLToPath(new URL('.', import.meta.url))`. When inlined into
the tsup bundle at `/app/dist/index.js`, this resolves to
`join('/app/dist', '..', 'locales')` = `/app/locales` — a path not present in the runtime image.

### Dockerfile change (`apps/api/Dockerfile`)

Added one COPY line in the `runtime` stage, after the existing `COPY --from=builder /app/prod /app`:

```dockerfile
# email-templates locales are loaded at runtime via readdirSync relative to the
# bundle location: join(fileURLToPath(new URL('.', import.meta.url)), '..', 'locales')
# = join('/app/dist', '..', 'locales') = /app/locales — copy them explicitly.
COPY --from=builder --chown=rovenue:rovenue /app/packages/email-templates/locales /app/locales
```

The builder stage has these files because `COPY packages ./packages` copies the full packages
directory at `/app/packages/email-templates/locales/{en,tr}/`. The destination `/app/locales`
is a sibling of `/app/dist/`, matching the runtime path computation exactly.
