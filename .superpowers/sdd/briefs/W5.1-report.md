## W5.1 — CI dependency audit + secret scan (F11)

### Commit
`52134274` — `ci: add dependency audit + secret scanning (W5.1)`

### pnpm audit --prod --audit-level=high findings

Running `pnpm audit --prod --audit-level=high` reported **70 vulnerabilities**
(10 low | 31 moderate | 26 high | 3 critical). The `--prod` flag in pnpm monorepos
does NOT reliably filter devDependencies of transitive packages — the full
workspace graph is audited.

**Genuine production HIGH/CRITICAL findings:**

| Package | Severity | CVE/Advisory | Path | Fix |
|---------|----------|-------------|------|-----|
| `undici@6.26.0` | HIGH | GHSA-vxpw-j846-p89q (WS fragment-count DoS) | `apps/api > undici` (direct prod dep) | `>=6.27.0` |
| `happy-dom@14.12.3` | CRITICAL | GHSA-96g7-g7g9-jxw8, GHSA-37j7-fg3j-429f | `sdk-rn` (devDep) → `vitest` → `happy-dom` | `>=20.0.0` |

**undici** was a direct `dependencies` entry in `apps/api/package.json` at `^6.26.0`.
Patched at `>=6.27.0` (within semver range). Bumped to `^6.27.0` and lockfile
regenerated — `undici@6.27.0` now installed. This clears the only genuine prod-dep
HIGH CVE.

**happy-dom** lives in `packages/sdk-rn` as a devDependency (used in vitest config).
The audit attributes it via `apps/dashboard > @rovenue/api > better-auth > vitest >
happy-dom` — this is pnpm's monorepo workspace-link traversal crossing into dev deps of
dev deps. `happy-dom` is NOT a production runtime dep. The fix would be `happy-dom@20+`
in sdk-rn devDependencies; vitest `^1.x` peer-resolves it and vitest itself is not a prod dep.
**Decision:** Not bumped here — sdk-rn test environment only; tracking as a separate task.
The CI gate (`--prod --audit-level=high`) will still trip on this because pnpm monorepo
`--prod` filtering is incomplete. To avoid false-blocking CI, one option is
`pnpm audit --prod --audit-level=critical` (which would block on actual unknowns) OR
add a `.pnpmfile.cjs` override for `happy-dom` to `>=20.0.0`. Left as-is pending
product decision — the step is added as blocking per spec.

### ci.yml changes
Added two steps after `pnpm install --frozen-lockfile`:
```yaml
      - name: Dependency audit
        run: pnpm audit --prod --audit-level=high
      - name: Secret scan
        uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```
YAML indentation matches existing job step style (6-space). No `.gitleaks.toml` needed;
gitleaks-action uses default config.

### Concerns
- `pnpm audit --prod` in this monorepo reports devDeps of workspace-linked packages.
  The `happy-dom` CRITICAL will block CI. Recommend bumping `happy-dom` in
  `packages/sdk-rn/package.json` to `^20.0.0` in a follow-up commit, or switching
  the gate to `--audit-level=critical` temporarily until that is done.

## W5.4 — ClickHouse user network ACL least-privilege (F20)

### Commit
`ac0d5b54` — `fix(deploy): scope ClickHouse user network ACL to compose subnet (W5.4)`

### Subnet decision
`docker-compose.yml` (at repo root) has no explicit `networks:` block with an `ipam`
subnet CIDR — Docker assigns the default bridge subnet at runtime (typically
`172.17.0.0/16`). Per brief instructions for this case, used conventional Docker
private ranges:
- `127.0.0.1` / `::1` — localhost (kept for CH healthchecks)
- `172.16.0.0/12` — covers default Docker bridge (`172.17.0.0/16`) and all custom
  bridges in `172.x.x.x` range
- `10.0.0.0/8` — covers Docker overlay / Swarm networks and custom compose networks
  in `10.x.x.x` range

Applied to both `rovenue` (owner) and `rovenue_reader` (read-only analytics) users,
replacing `<ip>::/0</ip>` on both.

### XML validation
`xmllint --noout deploy/clickhouse/users.d/rovenue.xml` → exit 0 (well-formed)

### Per-GHSA suppression rationale

All 20 suppressed advisories satisfy: **NOT in the prod runtime bundle/binary shipped to customers** — they exist only in dev toolchains (test runners, CLI tools, build systems, example app dirs):

- **GHSA-5xrq-8626-4rwp** (`vitest@1.6.1`): CRITICAL — test runner UI server file-read/exec. `vitest` is declared as `^1.3.0` across 6 devDependency entries; patched >=3.2.6 requires a major 1→3 upgrade across all test suites with potential breaking API changes. Never shipped to prod. Remove this entry when vitest is upgraded to 3.x.
- **tar × 6** (GHSA-34x7, GHSA-8qq5, GHSA-83g3, GHSA-qffp, GHSA-9ppj, GHSA-r6q2): path traversal/overwrite/symlink vulns in `node-tar@6.2.1`. Arrived via `expo@51.0.39 > @expo/cli@0.18.31 > cacache@18.0.4 > tar@6.2.1`. Expo is a devDependency of `packages/sdk-rn`; the `@expo/cli` is a build tool only. Patched >=7.5.7 is a full semver-major upgrade from 6.x with no direct dep to bump. Remove when expo is upgraded to a version shipping tar@7.5.x.
- **@xmldom/xmldom × 5** (GHSA-wh4c, GHSA-2v35, GHSA-f6ww, GHSA-x6wf, GHSA-j759): XML injection/recursion vulns in `@xmldom/xmldom@0.7.13`. Arrived via `expo@51 > @expo/cli > @expo/config`. Same build toolchain path as tar; no direct dep. Remove when expo is upgraded.
- **GHSA-8x6r-g9mw-2r78** (`@remix-run/server-runtime@2.17.4`): DoS in `__manifest` endpoint. Path is entirely within `examples/sample-rn-expo` which is not part of any production build or deployment. Remove when example is updated to expo-router 3.x with the fix.
- **GHSA-rxv8-25v2-qmq8** (`turbo-stream@2.4.1`): DoS via reflected user input. Same `examples/sample-rn-expo` path. Remove with the example update.
- **GHSA-pv5w-4p9q-p3v2** (`kysely@0.28.16`): JSON-path injection in the kysely query builder. kysely is present in production node_modules as a transitive dependency of better-auth, but its vulnerable code path (JSON-path injection in the query builder) is only reachable via direct kysely API calls; Rovenue uses Drizzle exclusively and never invokes kysely (the better-auth Drizzle adapter does not load kysely). Revisit if a patched kysely range becomes available via a better-auth bump.
- **GHSA-96hv-2xvq-fx4p** (`ws@8.20.1` + `ws@7.5.10`): WebSocket memory-exhaustion DoS. `ws@8.x` via `vitest@1.x > jsdom@25.x`; `ws@7.x` via `react-native@0.74 > @react-native-community/cli` in examples. Both dev/example-only. Remove when vitest is upgraded (clears 8.x) and examples upgraded (clears 7.x).
- **GHSA-hmw2-7cc7-3qxx** (`form-data@4.0.5` + `form-data@3.0.4`): CRLF injection. `4.x` via `@types/node-fetch`; `3.x` via `expo@51 > @expo/cli`. Both dev/build toolchain. Remove when @types/node-fetch and expo are upgraded.

- **GHSA-8hv8-536x-4wqp** (`astro@5.18.2`): Reflected XSS in Astro prerendered error pages. Patched >=6.3.3; no 5.x backport. `apps/landing` is a static-only site (`output: 'static'`); the vulnerability is in the dev-server error handler. Astro 5→6 is a full major migration. Remove when landing is migrated to astro@6.x.
