## W5.4 â€” ClickHouse user network ACL least-privilege (F20)

### Commit
`ac0d5b54` â€” `fix(deploy): scope ClickHouse user network ACL to compose subnet (W5.4)`

### Subnet decision
`docker-compose.yml` (at repo root) has no explicit `networks:` block with an `ipam`
subnet CIDR â€” Docker assigns the default bridge subnet at runtime (typically
`172.17.0.0/16`). Per brief instructions for this case, used conventional Docker
private ranges:
- `127.0.0.1` / `::1` â€” localhost (kept for CH healthchecks)
- `172.16.0.0/12` â€” covers default Docker bridge (`172.17.0.0/16`) and all custom
  bridges in `172.x.x.x` range
- `10.0.0.0/8` â€” covers Docker overlay / Swarm networks and custom compose networks
  in `10.x.x.x` range

Applied to both `rovenue` (owner) and `rovenue_reader` (read-only analytics) users,
replacing `<ip>::/0</ip>` on both.

## W5 Code-Review Follow-up â€” Fix 1 (Dockerfile devDep strip) + Fix 2 (kysely rationale)

**Date:** 2026-06-21  
**Triggering review:** Important findings on W5 (F12 attack-surface goal + kysely rationale precision).

### Fix 1 â€” Remove devDependencies from production Docker runtime image

**Problem:** The previous Dockerfile (`COPY --from=builder /app /app`) shipped ALL workspace node_modules to the runtime image, including tsx, vitest, testcontainers, and typescript â€” widening attack surface contrary to the F12 goal. The `CMD` correctly ran `node` (not tsx), but the packages remained on disk.

**Root cause of previous approach's failure:** `pnpm prune --prod` in a pnpm workspace removes root-level devDeps but keeps devDeps of all workspace packages (sdk-rn has vitest/tsx; api has tsx/vitest/testcontainers). The prune in the builder also lacked the pnpm store mount so it silently did nothing (0.4s). Neither approach worked reliably.

**Fix applied:** Restructured to use `pnpm deploy --prod`:
1. `deps` stage: `pnpm install --frozen-lockfile --ignore-scripts` (full workspace install, `--ignore-scripts` prevents fumadocs-mdx and other workspace postinstalls that need source files not copied in this stage)
2. `builder` stage: compile workspace packages (`@rovenue/shared`, `@rovenue/db`, `@rovenue/email-templates`) + API (`@rovenue/api`), then patch those workspace packages' `package.json` to redirect `main`/`exports` from `src/*.ts` â†’ `dist/*.js` so that bare `node` (not `tsx`) can load them at runtime
3. `pnpm --filter @rovenue/api deploy --prod --ignore-scripts /app/prod`: creates self-contained directory with ONLY production node_modules (707 packages vs 2477 in full install)
4. Post-deploy: `rm -rf /app/prod/node_modules/vitest` â€” removes vitest which pnpm deploy included because better-auth declares it as an optional peer dep (it is NOT imported by the API at runtime)
5. `runtime` stage: `COPY --from=builder /app/prod /app` â€” only the lean prod output

Also upgraded base image `node:20-alpine` â†’ `node:22-alpine` (satisfies `engines.node >=22.7` in package.json; the prior node:20 image caused `ERR_PNPM_UNSUPPORTED_ENGINE` failures). Pinned pnpm to 9.15.9 matching `packageManager` field.
