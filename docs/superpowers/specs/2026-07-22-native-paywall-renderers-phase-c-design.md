# Native Paywall Renderers — Phase C Design

Date: 2026-07-22
Status: approved (continuation of Phase B, `2026-07-22-paywall-builder-phase-b-design.md`)

## Overview

Phase C makes builder paywalls render **natively**: the Rust core carries
`builderConfig` to the façades, and Swift (SwiftUI) and Kotlin (Android
Views) renderers draw the same 7-node component tree the web renderer draws —
with **real store prices** feeding the `{{price}}` variables. Rendering is
100% native (no webview), matching RevenueCat/Adapty.

### Key decisions

1. **Core passes `builderConfig` as a raw JSON string** (`builder_config_json`
   on `CorePaywall`, like `remote_config_json`) — no uniffi modeling of the
   recursive tree. Each platform decodes into its own node model
   (Swift `Codable`, Kotlin `kotlinx-serialization` — already a dependency).
2. **Cross-platform parser contract via shared fixtures** (the
   bucketing-vectors pattern): `packages/shared/src/paywall/render-fixtures.json`
   holds accept/reject configs and variable-resolution vectors generated from
   the TS implementation (the source of truth). Vitest, Swift tests, and
   Kotlin tests all assert against the same file.
3. **Kotlin renders with Android Views, not Compose.** The SDK is a single
   Gradle module on Kotlin 1.9.24; adding Compose would force the Compose
   runtime and compiler-version coupling onto every consumer. Views cost more
   code but zero dependencies (minSdk 24 safe). A Compose wrapper module is
   an explicit future follow-up, not Phase C.
4. **Renderer semantics mirror the web renderer exactly** (the normative
   sibling): unknown node type → render `fallback` else nothing, never throw;
   theme pairs resolved by the platform's current light/dark mode (overridable);
   empty `packageIds` = all offering packages; selection init
   `defaultSelected ?? effectiveIds[0] ?? null`; purchase disabled on null
   selection; restore hidden when no handler; variables cell-scoped inside
   package cells, selected-scoped elsewhere.
5. **Real prices**: `PackageView` (packageName/price/pricePerPeriod/period)
   is derived from the platform's hydrated store product
   (StoreKit/Play Billing `priceString`, subscription period) by a
   deterministic pure mapper per platform, spec'd identically:
   `packageName = displayName`, `price = priceString ?? ""`,
   `period` = localized-ish unit label from the subscription period
   (`"month"`, `"year"`, `"week"`, `"day"`, empty for non-subs),
   `pricePerPeriod = price + "/" + period` (or just `price` when period empty).
6. **Builder paywalls auto-track**: the native renderers call the existing
   `logPaywallShown(paywall)` once on first appearance (Adapty parity —
   remote-config paywalls stay manual).
7. **Purchase wiring is internal**: the renderer maps the selected package
   identifier to the offering package / store product and calls the SDK's
   existing purchase API; success invokes an `onPurchaseCompleted` callback
   (host dismisses), failure surfaces via `onPurchaseFailed` (paywall stays).
8. **RN is a non-goal this phase** except DTO passthrough: `PaywallDTO`
   gains `builderConfigJson` so JS can read it; the native *presentation*
   bridge (modal host of the Swift/Kotlin renderers) is a follow-up that
   requires a simulator verification session (see memory M7 pod caveats).

## 1. Core + façade plumbing

- `PaywallWire` gains `builder_config: Option<serde_json::Value>` (serde
  default — absent field tolerated, existing cache entries keep decoding).
- `CorePaywall` gains `builder_config_json: Option<String>` (re-serialized
  verbatim) — UDL `string?`.
- Façades: Swift/Kotlin `Paywall` types gain `builderConfigJson: String?`
  (raw). RN `PaywallDTO` + mapped `Paywall` gain `builderConfigJson` /
  `builderConfig` (JS parses like remoteConfig). No behavior change.

## 2. Shared fixtures (`render-fixtures.json`)

```jsonc
{
  "accept": [ { "name": "...", "config": { ...BuilderConfig... } } ],   // ≥6: every node type, fallback nodes, theme pairs, multi-locale
  "reject": [ { "name": "...", "config": { ... } } ],                    // ≥5: bad axis, missing id, bad formatVersion, malformed fallback child, non-object localization
  "variables": [ { "text": "...", "pkg": { ...PackageView... } | null, "expected": "..." } ],  // reuse/extend the TS resolveVariables cases
  "resolveText": [ { "locale": "...", "key": "...", "expected": "..." | null } ]               // against accept[0]'s config
}
```

Generated by a script executing the real TS schema/helpers; frozen
thereafter. Swift loads it `#filePath`-relative; Kotlin via a repo-relative
`File` path from the Gradle project dir; Vitest asserts the TS side accepts/
rejects identically (guards fixture rot).

## 3. Swift renderer (SwiftUI, iOS 16+/macOS 12)

- `Sources/Rovenue/PaywallUI/` — `BuilderConfigModel.swift` (Codable node
  tree mirroring the schema; unknown `type` decodes into a `.unknown(fallback:)`
  case rather than throwing), `PackageViewMapping.swift` (pure mapper per
  decision 5), `RovenuePaywallView.swift` (SwiftUI):

```swift
public struct RovenuePaywallView: View {
  public init(
    paywall: Paywall,                                // must carry builderConfigJson; else renders EmptyView
    locale: String? = nil,
    colorSchemeOverride: ColorScheme? = nil,          // else follows environment
    onPurchaseCompleted: ((PurchaseResult) -> Void)? = nil,
    onPurchaseFailed: ((Error) -> Void)? = nil,
    onClose: (() -> Void)? = nil,
    onRestore: (() -> Void)? = nil,                   // nil → restore buttons hidden
    onUrl: ((URL) -> Void)? = nil                     // nil → url buttons inert; renderer NEVER opens URLs itself
  )
}
```

- Internal: selection `@State`, purchase via the SDK's existing purchase
  entry point for a package (locate the exact API), auto `logPaywallShown`
  in `onAppear` (once). Stack v/h/z → VStack/HStack/ZStack; theme pairs via
  effective color scheme; text via resolveText+resolveVariables ports
  (fixture-tested).

## 4. Kotlin renderer (Android Views, existing module)

- `src/main/kotlin/dev/rovenue/sdk/paywallui/` — `BuilderConfigModel.kt`
  (kotlinx-serialization, unknown type → fallback semantics),
  `PackageViewMapping.kt` (pure), `PaywallViewRenderer.kt`:

```kotlin
class RovenuePaywallView(context: Context) : FrameLayout {
  fun bind(paywall: Paywall, options: PaywallViewOptions)   // renders the tree; no-op + empty when builderConfigJson null
}
data class PaywallViewOptions(
  val locale: String? = null,
  val darkMode: Boolean? = null,                     // null → follow configuration
  val onPurchaseCompleted: ((PurchaseResult) -> Unit)? = null,
  val onPurchaseFailed: ((Throwable) -> Unit)? = null,
  val onClose: (() -> Unit)? = null,
  val onRestore: (() -> Unit)? = null,               // null → restore hidden
  val onUrl: ((String) -> Unit)? = null,
)
```

- Stacks → LinearLayout (v/h) / FrameLayout (z); text/buttons standard
  widgets styled programmatically; purchase via the SDK's existing
  `purchase(activity, product, option)` path (activity obtained from the
  view's context, error if none); auto `logPaywallShown` on first attach.
- Tests: parser + mapping + resolve logic pure-JVM against fixtures
  (`testDebugUnitTest`); view behavior smoke via Robolectric ONLY if the
  module already has it — otherwise pure logic tests + a documented manual
  smoke (do not add new test infra).

## 5. Testing

- Fixture round-trip on all three: Vitest (guards the fixture), Swift
  `swift test`, Kotlin `testDebugUnitTest` — every accept parses, every
  reject fails, every variable/resolveText vector matches.
- Renderer behavior tests per platform where infra exists (SwiftUI via
  ViewInspector is NOT a current dependency — test the view MODEL layer:
  extract selection/purchase-enable/text-resolution into pure testable
  helpers, keep SwiftUI bodies thin; same principle for Kotlin Views).
- `cargo test` + regen bindings clean; full build stays green.

## Non-goals (Phase C)

- Compose renderer/module; RN native presentation bridge (follow-up w/
  simulator session); watchOS/tvOS/macOS-native chrome beyond what SwiftUI
  gives for free.
- Image remote-loading/caching layers beyond the platform default
  (AsyncImage on iOS; a minimal URL loader on Android without adding Coil).
- Conditions/overrides, countdowns, video nodes — schema unchanged.
- Paywall presentation convenience APIs (presentPaywallIfNeeded etc.) —
  hosts embed the view; helpers can come later.
- Fallback bundled files, min-SDK-version targeting rules.
