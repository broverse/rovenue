Rovenue covers the same subscription-management use cases as RevenueCat but runs on your own infrastructure. This page maps familiar RevenueCat concepts to their Rovenue equivalents so you can orient quickly.

## Concept mapping

| RevenueCat | Rovenue |
|---|---|
| `Purchases.configure(apiKey)` | `Rovenue.configure({ apiKey })` — `baseUrl` is optional (self-hosting only) |
| `getCustomerInfo().entitlements` | `entitlement(id)` / `entitlementsAll()` |
| `logIn(appUserID)` | `identify(appUserId)` — server-side bind + auto-transfer of prior installs |
| `Purchases.getOfferings()` | `Rovenue.getOfferings()` |
| SDK-driven `purchase(package)` | `Rovenue.purchase(pkg)` — same pattern; `getOfferings()` then `purchase(pkg)` |
| `addCustomerInfoUpdateListener` | `addChangeListener` (RN) / `changes` AsyncStream (Swift) / SharedFlow (Kotlin) |

## Purchase flow

The purchase flow in Rovenue mirrors RevenueCat's: call `getOfferings()` to
fetch your configured products, then pass the chosen package to `purchase(pkg)`.
The SDK triggers the native store sheet (StoreKit 2 on iOS, Play Billing on
Android), validates the receipt server-side, and resolves with updated
entitlements and credit balance — no separate receipt-posting step required.

```ts
const offerings = await Rovenue.getOfferings();
const pkg = offerings.current?.packages[0];

if (pkg) {
  try {
    const result = await Rovenue.purchase(pkg);
    // result.entitlements / result.virtualCurrencies are already up to date
  } catch (e) {
    if (e instanceof PurchaseCancelledError) { /* user dismissed — ignore */ }
    else if (e instanceof PurchasePendingError) { /* Ask-to-Buy / deferred */ }
    else throw e;
  }
}

await Rovenue.restorePurchases();
```

## Key differences from RevenueCat

- **Self-hosted.** Rovenue runs on your own infrastructure — no revenue share,
  no third-party data processing. You own the database and control retention.
- **Virtual currencies built-in.** Rovenue ships a first-class multi-currency ledger for consumable products, loyalty systems, and metered billing. Balances are read client-side; spending is a server-side operation via `POST /v1/virtual-currencies/:appUserId/:code/transactions`.
- **Identity model.** `identify(appUserId)` performs a server-side bind and automatically transfers any prior install's purchases and credits to the current device (similar to RevenueCat's "transfer to new ID" behaviour). Call `logOut()` on sign-out to mint a fresh `rovenueId`. Bulk or programmatic merges are available via the secret-key backend API.
- **Open source (AGPLv3).** You can read, audit, and fork the full stack.
