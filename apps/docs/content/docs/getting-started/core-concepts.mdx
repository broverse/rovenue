The Rovenue SDK is built around a small set of concepts that apply identically across React Native, Swift, and Kotlin. Understanding them makes every other part of the API predictable.

---

## The two user IDs: `rovenueId` and `appUserId`

Every install gets a **permanent, anonymous** `rovenueId` (prefix `rov_`) the moment `configure()` is called. It is stable for the lifetime of the app install — it does not rotate on app restart, network change, or OS update.

At login, you call `identify(appUserId)` to attach your own backend user ID to the `rovenueId`. From that point, Rovenue uses the `appUserId` as the canonical subscriber identity.

```
Install                   Login                       Sign-out
  │                          │                            │
  ▼                          ▼                            ▼
rovenueId = rov_abc...   appUserId = "usr_123"        logOut()
(anonymous subscriber)   (identified subscriber)      (new rovenueId minted)
```

The current state is always available via `currentUser()`:

### Types on this page

```ts
type User = {
  rovenueId: string;        // permanent per-install id, prefix rov_; never changes
  appUserId: string | null; // your backend user id; null until identify()
};

type ChangeEvent =
  | 'EntitlementsChanged'   // active entitlements updated
  | 'IdentityChanged'       // rovenueId or appUserId changed (identify / logOut)
  | 'CreditBalanceChanged'; // credit balance changed
```

| Field | Type | Description |
|-------|------|-------------|
| `rovenueId` | `string` | Permanent per-install identifier. Never reassigned for the life of the install. |
| `appUserId` | `string \| null` | Your backend's user ID. `null` in anonymous mode. Set by `identify()`; cleared by `logOut()`. |

> Swift uses `User.rovenueId` / `User.appUserId`. Kotlin mirrors the same field names. `ChangeEvent` is a Swift enum (`.entitlementsChanged`) and a Kotlin enum (`ENTITLEMENTS_CHANGED`). See [Types](/docs/reference/types#user).

<Callout type="info">
The `rovenueId` is the unit of billing: it is what gets linked to App Store / Play Store receipts and what holds the credit balance. Even in anonymous mode, purchases work correctly — they attach to the `rovenueId` and are transferred to the `appUserId` automatically when `identify()` is called.
</Callout>

### `identify()` — binding a logged-in user

Call `identify(appUserId)` once your auth flow completes and you have a stable, permanent user ID. The call is **optimistic**: local state (including `currentUser()`) updates synchronously; in the background the SDK calls `POST /v1/identify` to perform the server-side bind and purchase transfer.

<Callout type="warn">
Pass a **stable, permanent** user ID — a UUID or database primary key from your backend. Never pass a session token, email address, or anything that can change. Passing an unstable ID creates a new subscriber record per login and the merge cannot be undone. See [Identifying users](/docs/guides/identifying-users) and [Identity & consent](/docs/guides/identity-consent).
</Callout>

### `logOut()` — clearing identity on sign-out

`logOut()` mints a fresh `rovenueId` and clears the `appUserId`. All scope-bound caches (entitlements, credits, offerings) are cleared. The next user on a shared device starts completely clean. No network call is needed — the operation is local-only.

<Callout type="warn">
Always call `logOut()` on sign-out, especially on shared devices (family iPad, demo devices). Failing to do so leaves the previous user's entitlements accessible to the next session.
</Callout>

Method references: [`identify`](/docs/reference/methods#identity), [`logOut`](/docs/reference/methods#logout), [`currentUser`](/docs/reference/methods#currentuser).

## Entitlements — what your app actually checks

An **entitlement** is a named access grant your app checks at runtime (for example, `"pro"`, `"premium"`, or `"credits_pack"`). Entitlements are independent of store SKUs: you configure the mapping in the Rovenue dashboard, then check the entitlement ID in code.

<Callout type="info">
**The key mental model:** your feature-gating code never references a product ID or SKU. It only ever asks "does the current user have entitlement `"pro"`?" The Rovenue dashboard is where you wire store products to entitlements. This means you can add new products, change prices, or run a promo without shipping a new binary.
</Callout>

Entitlements have an `isActive` boolean — the only field your app needs at the gate:

```ts
type Entitlement = {
  id: string;                // entitlement id as configured in the dashboard
  isActive: boolean;         // true = access currently granted
  productIdentifier: string; // store SKU that created the grant
  store: string;             // "app_store" | "play_store" | "stripe"
  expiresIso?: string;       // ISO-8601 expiry; undefined for lifetime products
};
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | Dashboard entitlement ID. This is what you pass to `entitlement()`. |
| `isActive` | `boolean` | The only field to check at the gate. |
| `productIdentifier` | `string` | The store SKU that created the grant. |
| `store` | `string` | Which store issued the grant. |
| `expiresIso` | `string?` | Expiry in ISO-8601. Absent for lifetime products. |

> See [Types](/docs/reference/types#entitlement) and [Entitlements guide](/docs/guides/entitlements).

<Callout type="warn">
Check `isActive`, not `expiresIso`. An entitlement with a future expiry date and `isActive: false` is in a grace period or was revoked. The server sets `isActive` based on subscription state — your app should treat it as a boolean gate.
</Callout>

## Credits — integer balance, server-deducted

Credits are an integer balance associated with the current subscriber. The SDK exposes them as a cache-first read (`creditBalance()` / `useCreditBalance()`). Deduction happens server-side via `consumeCredits()`.

```ts
// Read (cache-first)
Rovenue.creditBalance()    // number

// Deduct (server-side; returns new balance)
Rovenue.consumeCredits(amount: number, description?: string) // number
```

Every `consumeCredits` call includes an idempotency key generated by the SDK. If the network drops after the server deducts but before the client receives the response, retrying the call does not double-deduct.

<Callout type="info">
Credit balance is part of the `PurchaseResult` — after a credit pack purchase, `result.creditBalance` already reflects the newly added credits. No separate refresh is needed. See [Credits guide](/docs/guides/credits).
</Callout>

## Offerings and packages — the product catalogue

An **offering** is a named, versioned slice of your product catalogue as configured in the Rovenue dashboard. The `current` offering is what you show on the paywall for this user (you can A/B test offerings with experiments). Each offering contains **packages** — individual purchasable items.

```ts
type Offerings = {
  current: Offering | null;
  all: Record<string, Offering>;
};

type Offering  = { identifier: string; isDefault: boolean; packages: Package[] };
type Package   = { identifier: string; product: StoreProduct };
type StoreProduct = {
  id: string;
  type: 'subscription' | 'consumable' | 'non_consumable';
  displayName: string;
  priceString: string | null; // localised, e.g. "$4.99"
  price: number | null;
  currencyCode: string | null;
};
```

`getOfferings()` returns a cached result and refreshes in the background. When you need a guaranteed-fresh list (unusual; typically not needed on the critical path), call `refreshEntitlements()` and then `getOfferings()`.

## Cache-first reads and background sync

All read operations — `entitlement()`, `entitlementsAll()`, `creditBalance()`, `getOfferings()` — return instantly from an on-device SQLite cache powered by the Rust core. They **never block on a network call**.

The SDK keeps the cache warm automatically:

1. On `configure()`: loads the SQLite snapshot, serves it immediately.
2. While foregrounded: polls the server on a configurable interval (default: 30 s) and applies deltas silently.
3. On purchase: immediately calls `refreshEntitlements()` before resolving `purchase()`, so `PurchaseResult` is always authoritative.
4. On `setForeground(true)`: forces an immediate refresh (call this from your app's foreground lifecycle event).

<Callout type="info">
The cache is backed by rusqlite — it persists across app restarts. On first launch (empty cache) entitlements are `null` / empty until the first network poll completes. Design your paywall to handle this gracefully (show a loading state, not a blank screen).
</Callout>

## The change stream — reacting to updates

When the SDK receives updated data from the server, it emits a `ChangeEvent`. Subscribe once and update your UI reactively — no polling, no manual refresh.

```ts
type ChangeEvent =
  | 'EntitlementsChanged'   // active entitlements updated
  | 'IdentityChanged'       // rovenueId or appUserId changed
  | 'CreditBalanceChanged'; // credit balance changed
```

| Event | When it fires |
|-------|---------------|
| `EntitlementsChanged` | After purchase, restore, server-side revocation, or background poll. |
| `IdentityChanged` | After `identify()` or `logOut()`. |
| `CreditBalanceChanged` | After a purchase that grants credits or a `consumeCredits` call. |

<Tabs groupId="platform" persist items={['React Native', 'Swift', 'Kotlin']}>
  <Tab value="React Native">
    ```ts
    import { Rovenue } from '@rovenue/react-native-sdk';
    import { useEffect } from 'react';

    function useRovenueChanges() {
      useEffect(() => {
        const unsubscribe = Rovenue.addChangeListener(event => {
          if (event === 'EntitlementsChanged') {
            console.log('Entitlements updated');
          }
          if (event === 'CreditBalanceChanged') {
            console.log('New balance:', Rovenue.creditBalance());
          }
        });
        return unsubscribe; // cleans up on unmount
      }, []);
    }
    ```

    In practice, using `useEntitlement()` and `useCreditBalance()` hooks is simpler — they subscribe to the change stream internally and re-render automatically.
  </Tab>
  <Tab value="Swift">
    Swift exposes `ChangeEvent` as a typed enum and the stream as `AsyncStream<ChangeEvent>`:

    ```swift
    import Rovenue

    Task {
      for await event in Rovenue.shared.changes {
        switch event {
        case .entitlementsChanged:
          // read new entitlements
        case .creditBalanceChanged:
          let balance = await Rovenue.shared.creditBalance()
          print("New balance:", balance)
        case .identityChanged:
          let user = Rovenue.shared.currentUser()
          print("Now identified as:", user.appUserId ?? "anonymous")
        }
      }
    }
    ```
  </Tab>
  <Tab value="Kotlin">
    Kotlin exposes `ChangeEvent` as a typed enum and the stream as `SharedFlow<ChangeEvent>`:

    ```kotlin
    import dev.rovenue.sdk.Rovenue
    import dev.rovenue.sdk.ChangeEvent

    viewModelScope.launch {
      Rovenue.shared.changes.collect { event ->
        when (event) {
          ChangeEvent.ENTITLEMENTS_CHANGED -> {
            // read new entitlements
          }
          ChangeEvent.CREDIT_BALANCE_CHANGED -> {
            val balance = Rovenue.shared.creditBalance()
            println("New balance: $balance")
          }
          ChangeEvent.IDENTITY_CHANGED -> {
            val user = Rovenue.shared.currentUser()
            println("Now identified as: ${user.appUserId ?: "anonymous"}")
          }
        }
      }
    }
    ```
  </Tab>
</Tabs>

## `logOut()` — the shared-device contract

When a user signs out, call `logOut()`. This:

1. Mints a fresh `rovenueId` (the old one is preserved on the server, attached to the `appUserId`).
2. Clears `appUserId` from local state.
3. Clears all scope-bound caches (entitlements, credits, offerings).
4. Emits `IdentityChanged`.

The next user who opens the app starts in a clean anonymous session. Their subsequent `identify()` call attaches them to their own subscriber record.

<Callout type="warn">
Skipping `logOut()` on a shared device means the next user sees the previous user's entitlements until the background poll overwrites the cache — up to 30 seconds. Always call it on sign-out.
</Callout>

Method reference: [`logOut`](/docs/reference/methods#logout).
