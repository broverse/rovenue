The Rovenue SDK is built around a small set of concepts that apply across React Native, Swift, and Kotlin. Understanding them makes the rest of the API predictable.

## Anonymous vs known user

Every install starts in an anonymous state. The SDK generates a stable `rovenueId` on first launch (prefix `rov_`) and uses it to track events, entitlements, and credits before you know who the user is. The `rovenueId` is permanent for the lifetime of the install.

Call `identify(appUserId)` once you have an app-side identity (after login, for example). This is a **server-side bind**: Rovenue attaches your `appUserId` to the current `rovenueId` and automatically transfers any prior install's purchases and credits to this device. The call is optimistic — local state updates immediately.

On user sign-out, call `logOut()`. This mints a fresh `rovenueId` and clears the `appUserId` so the next user on a shared device starts clean. See [Identifying users](/docs/guides/identifying-users) for the full flow.

## Entitlement vs product

An **entitlement** is an access grant your app checks at runtime (for example, `"pro"` or `"credits_pack_100"`). Store products — App Store subscriptions, Play Store products, or Stripe prices — are mapped to entitlements in the Rovenue dashboard.

Your app should always check entitlements, not product IDs. This keeps your feature-gating code decoupled from store-specific SKUs and lets you remap products (for example during a pricing change) without shipping a new binary.

## Cache-first reads and background refresh

All read operations — `entitlement()`, `entitlementsAll()`, `creditBalance()`, and their hook equivalents — return instantly from an on-device SQLite cache. They never block on a network call.

The SDK keeps the cache warm automatically: while the app is in the foreground it polls the server and applies updates in the background. Your UI reacts to the change without doing anything. When you need an authoritative up-to-date snapshot immediately (for example, right after a purchase), call `refreshEntitlements()` explicitly — that method contacts the server before resolving.

## Credits

Credits are an integer balance associated with a subscriber. The SDK exposes `creditBalance()` (or `useCreditBalance()` in React) as a cache-first read.

Deducting credits is done server-side via `consumeCredits`. The SDK sends an `Idempotency-Key` with every consume call, so retrying after a network failure does not double-deduct the balance.

## SDK-driven purchases

<Callout type="info">
The SDK drives the full purchase lifecycle. Call `getOfferings()` to fetch your configured products, then `purchase(pkg)` to trigger the native store sheet (StoreKit 2 on iOS, Play Billing on Android) and validate the receipt server-side — all in one call.
</Callout>

`purchase(pkg)` resolves only after server-side validation succeeds and returns a `PurchaseResult` with up-to-date `entitlements` and `creditBalance`. Your app does not need to post receipts manually or poll for state changes. On success the SDK also emits a change event so any listening UI updates automatically.
