Every install has two identifiers:

- **`rovenueId`** — a permanent ID generated by the SDK on first launch. It is
  always present, never changes for the lifetime of the install, and is the
  stable identity to which purchases, credits, and entitlements are attached.
- **`appUserId`** — your backend's user ID, attached by calling `identify()`.
  It is optional: a user is anonymous until you call `identify`.

## Reading the current user

`currentUser()` is a **cache read** — it never hits the network and returns
instantly. It returns a `User` object:

```ts
type User = {
  rovenueId: string;    // always present
  appUserId: string | null; // null until identify() has been called
};
```

<Tabs groupId="platform" persist items={['React Native', 'Swift', 'Kotlin']}>
  <Tab value="React Native">
    ```ts
    import { Rovenue, useCurrentUser } from '@rovenue/react-native-sdk';

    // Imperative cache read
    const user = await Rovenue.currentUser();
    console.log(user.rovenueId, user.appUserId);

    // Reactive hook alternative
    function Header() {
      const user = useCurrentUser();
      return <Text>{user?.appUserId ?? '(anonymous)'}</Text>;
    }
    ```
  </Tab>
  <Tab value="Swift">
    ```swift
    let user = await Rovenue.shared.currentUser()
    print(user.rovenueId, user.appUserId ?? "(anonymous)")
    ```
  </Tab>
  <Tab value="Kotlin">
    ```kotlin
    val user = Rovenue.shared.currentUser()
    println("${user.rovenueId} / ${user.appUserId}")
    ```
  </Tab>
</Tabs>

## Identifying a user

Call `identify(appUserId)` after your user signs in. This call is
**server-side**: Rovenue binds the `appUserId` label to the current
`rovenueId` and, if that `appUserId` already exists on another install, it
**automatically transfers** the prior holder's purchases and credits to the
current device (similar to RevenueCat's "transfer to new ID" behaviour).

<Tabs groupId="platform" persist items={['React Native', 'Swift', 'Kotlin']}>
  <Tab value="React Native">
    ```ts
    // Call immediately after your auth flow completes
    await Rovenue.identify('user_abc123');
    ```
  </Tab>
  <Tab value="Swift">
    ```swift
    // Call immediately after your auth flow completes
    try await Rovenue.shared.identify("user_abc123")
    ```
  </Tab>
  <Tab value="Kotlin">
    ```kotlin
    // Call immediately after your auth flow completes
    Rovenue.shared.identify("user_abc123") // suspend
    ```
  </Tab>
</Tabs>

<Callout type="warn">
Use **opaque, unguessable** IDs for `appUserId` — never an email address,
username, or any value a third party could predict. Because `identify` can move
purchases between installs, a guessable ID is an account-takeover surface: an
attacker who knows (or can guess) a victim's ID can call `identify` on their
own device and claim the victim's entitlements.
</Callout>

## Signing out

On user sign-out, call `logOut()`. This mints a fresh `rovenueId` for the
install so the next user on a shared device starts with a clean slate and
cannot inherit the previous user's entitlements or credit balance.

## Backend-driven consolidation

For cases where you need authoritative, bulk, or programmatic merging (e.g.
importing subscribers from another platform), your **backend** can call the
transfer endpoint directly:

```
POST /v1/subscribers/transfer
Authorization: Bearer <secret-api-key>
```

<Callout type="warn">
Never put your **secret** API key (`rov_sec_…`) in the app. The transfer
endpoint is a server-to-server call only. Expose a lightweight endpoint in your
own backend that verifies the user session and then calls Rovenue on their
behalf.
</Callout>
